Practical Python Project Structure

Architecture • Python • January 15, 2025

Building maintainable Python applications requires thoughtful project organization. This guide covers practical patterns for structuring medium-sized services that balance clarity with scalability.

Core Principles

When structuring a Python project, consider these foundational principles:

  • Separation of Concerns - Keep business logic, data access, and presentation layers distinct
  • Import Clarity - Structure modules so imports are intuitive and dependencies are clear
  • Testability - Organize code to make unit and integration testing straightforward
  • Configuration Management - Centralize settings and environment-specific configuration

Recommended Structure

my_service/
├── src/
│   └── my_service/
│       ├── __init__.py
│       ├── main.py              # Application entry point
│       ├── config.py            # Configuration management
│       ├── models/              # Data models and schemas
│       │   ├── __init__.py
│       │   ├── user.py
│       │   └── product.py
│       ├── services/            # Business logic layer
│       │   ├── __init__.py
│       │   ├── user_service.py
│       │   └── product_service.py
│       ├── repositories/        # Data access layer
│       │   ├── __init__.py
│       │   ├── base.py
│       │   └── user_repository.py
│       └── api/                 # API routes and handlers
│           ├── __init__.py
│           ├── dependencies.py
│           └── v1/
│               ├── __init__.py
│               └── users.py
├── tests/
│   ├── __init__.py
│   ├── conftest.py
│   ├── unit/
│   └── integration/
├── docs/
├── requirements.txt
├── requirements-dev.txt
├── pyproject.toml
├── Dockerfile
└── README.md

Key Patterns

This structure follows several important patterns that scale well as your project grows:

Layered Architecture

The separation between models, services, repositories, and API layers creates clear boundaries. Each layer has a specific responsibility and depends only on layers below it.

Dependency Injection

Services depend on abstract interfaces rather than concrete implementations. This makes testing easier and keeps your business logic decoupled from infrastructure concerns.

class UserService:
    def __init__(self, user_repo: UserRepository):
        self._user_repo = user_repo

    async def create_user(self, user_data: UserCreate) -> User:
        # Business logic here
        return await self._user_repo.create(user_data)

Configuration Best Practices

Centralize configuration in a dedicated module that handles environment variables, validation, and defaults:

from pydantic import BaseSettings

class Settings(BaseSettings):
    database_url: str
    redis_url: str = "redis://localhost:6379"
    log_level: str = "INFO"

    class Config:
        env_file = ".env"

settings = Settings()

Testing Strategy

The structure naturally supports different testing approaches:

  • Unit tests for individual services and utilities
  • Integration tests for repository and API layers
  • End-to-end tests for complete user workflows

This organization has served me well across multiple Python services, from small APIs to larger distributed systems. The key is starting with clear boundaries and evolving the structure as your needs grow.
